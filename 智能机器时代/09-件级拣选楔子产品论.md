# 09 · 件级拣选楔子产品论(PM 学习路线 Stage 3 · capstone 范例)

> 这是 [08 PM学习路线](08-PM学习路线.md) 里 Stage 3 的产出物范例:挑一个场景(物流件级拣选),
> 把目标任务、要求的几个9、单位经济学、数据飞轮、形态、beachhead、**在位者**全写清楚。
> 量具用 [07 判断打分卡](07-判断打分卡.md);正反案例参 [Covariant](../02-companies/covariant.md) / [Chef](../02-companies/chef-robotics.md)。

---

## 0. 一句话定位
> **为「反亚马逊联盟」(3PL + 中型零售/品牌商)提供件级拣选 RaaS,beachhead 切「容错商品」(美妆个护/非生鲜日用),
> 在那里今天就能把可靠性做到工业级、把单位经济学做正,并用部署数据滚出泛化飞轮。**

## 1. 目标任务(收窄到不能再窄)
- **任务**:goods-to-person 拣选站,机械臂从料箱抓**单件商品**放入订单容器。
- **beachhead 商品集**:规整、可吸附、刚性、非生鲜——盒装/瓶装/泡罩/袋装的美妆个护、日用消费品。
- **明确排除**:可变形(衣物/线缆)、生鲜、易碎玻璃、透明、超重。
- **为什么这么窄**:这是「9」够得着的子集(泛化只需轴 1–3),失败可重试(容错),让产品**今天**能部署、能盈利,而不是等通用解。

## 2. 要求的可靠性『9』
- **部署门槛:单拣成功率 ≥ 99%(两个9)**——对应「1 监管员盯 ~10 台」,单位经济学才翻正。
- 真账本指标不是峰值成功率,是**接管率 / MTBI(平均多少拣一次人工干预)**。
- **失败设计**:可重试 N 次 → 仍失败 → 降级人工兜底(graceful)。这正是选「容错商品 + G2P 有人在场」的原因。

## 3. 单位经济学(承接 Stage 1)
| 可靠性 | 每拣成本 | vs 人工 $0.063 |
|---|---|---|
| 90% | $0.070 | ❌ 亏 |
| 99% | $0.035 | ✅ 省 44% |
| 99.9% | $0.032 | ✅ 省 49% |
- **临界点 ~95%**。产品的全部工程努力 = 把这条线从 95% 推到 99%+ 并稳住。
- **定价**:RaaS 按产出/订阅,客户零 CapEx;我们用设备融资出资车队、自背可靠性风险 → 倒逼把 9 做上去(Chef 模式)。
- 加分杠杆:机器人 20–24h 跑(人工 8–16h);旺季/招不到人时价值 = ∞。

## 4. 数据飞轮(护城河)
- 每个部署站 = **数据采集机**(Covariant 语)。
- 飞轮:更多部署 → 更多真实抓取轨迹(尤其失败案例)→ 对更多 SKU/姿态泛化 → 成功率↑ → 经济更好 → 更多客户 → 更多数据。
- RaaS 让我们持有车队、持续回流数据。
- **启动策略**:beachhead 商品窄 → 数据同质 → 快速到 99% → 证明 ROI → 用赚来的钱往「更难商品」扩(泛化轴往上爬)。

## 5. 形态选择(为什么机械臂+料箱,不是人形)
- **选**:固定式协作臂 + 3D 视觉 + 吸盘(主)/夹爪,嵌入现有 G2P 工作站。
- **不选人形**:任务在固定工位、料箱由 AMR 送到面前(移动已被解决),不需要移动/双足;人形是对「常有更便宜专用解」的过度设计(参 04-商业地图)。
- **BOM 驱动**:协作臂 + 3D 相机 + 末端执行器 + 集成;目标整站成本压到让 RaaS 时薪 < 人 + 摊销。
- **末端执行器是隐藏胜负手**:吸盘覆盖容错商品大部分;难抓的留给后续迭代。

## 6. 为什么是这个 beachhead(why here, why now)
- **劳动**:件级拣选是仓库「最后的人工孤岛」,劳动最密集、招工难、流动率高、旺季崩。
- **传统做不了**:高混合 SKU + 任意姿态,打死刚性自动化 → 给学习型机械臂留出车道。
- **容错**:失败便宜可重试 + G2P 有人兜底 → 2–3 个9 就能起步,不必等通用解。
- **技术成熟度**:这个子集的「9」今天够得着(Covariant/Radial 在 H&B 做到 ~425 拣/时已证明)。

## 7. 谁是在位者(最关键、最该诚实的一节)
> **在位者是亚马逊。** 这是件级拣选最危险的事实:它是亚马逊的核心业务;亚马逊自研 Sparrow/Blue Jay,且**逆向收购了 Covariant**——**数据飞轮挡得住创业对手,挡不住能收编你团队的巨头。**

**缓解策略(直接来自 Covariant vs Chef 的镜像):**
1. **服务「反亚马逊联盟」**:3PL + 中型零售/品牌商——正被亚马逊碾压、急需自动化、且永远不会是亚马逊。**我们卖给亚马逊的敌人。**
2. **往亚马逊不碰的合规/专业纵深走**(美妆个护 → 医药):监管 + 行业 know-how = 巨头懒得啃的壁垒。
3. **数据+客户关系飞轮快滚**:让公司「独立时比被收购时更值钱」,抬高被收编的代价。

**残余风险诚实承认**:若拣选模型对亚马逊自身仓库价值够大,acquihire 风险无法归零。纵深垂直 + 反亚马逊客户基,是把它压到最低、而非消除的解。

## 8. 打分卡收口
| 轴 | 评分 | 依据 |
|---|---|---|
| 泛化 | ★★★★☆ | beachhead 只需轴 1–3(容错商品) |
| 几个9 | ★★★★☆ | 容错任务 + 99% 目标 + 接管率兜底,工业级可达 |
| 数据效率 | ★★★★☆ | 窄商品集 + RaaS 飞轮 |
- **楔子判定:✅ 真能推进**(人招不到 + 传统做不了 + 失败便宜 + 专有数据)。
- **最大风险不在三轴,在在位者(亚马逊)**。解法:卖给它的敌人 + 往它不碰的纵深走。

## 9. 给自己的诚实结论(gut check)
- 这是个**能赚钱、能起步的真楔子**,不是讲故事。
- 但它的天花板与退出,被**亚马逊这个在位者深刻约束**。清醒的 PM 会二选一:
  - 接受「做一个被巨头环伺的高价值垂直 RaaS」,靠纵深 + 反亚马逊客户基求生;
  - 或换一个**没有科技巨头在位**的楔子(回看 Chef:食品制造就没有亚马逊)——这恰恰解释了**为什么 Chef 比 Covariant 活得好**。

> **一句话**:件级拣选是个真楔子,但选它,就是选了和亚马逊同场。要么用「卖给它的敌人 + 钻它不碰的纵深」活下去,要么——像 Chef 那样,去找一个没有巨头的场。
