# Freshness Sweep · 打法手册

> 一次「刷新扫描」的标准动作。任何会话、任何 agent、定时例行，都照这套跑——保证一致、可复现、不污染。

---

## 输入
- [watchlist.md](watchlist.md)（靶子 + 当前基线值）
- 当前日期（用真实当天日期，不要照搬旧日期）

## 步骤

1. **逐项搜**：对 watchlist 每一行，联网搜「对象名 + 关键词（融资/估值/IPO/订单/发布）+ 当年/latest」。
   优先级顺序：A(IPO) → B(估值) → C(订单) → D(技术) → E(上游) → F(格局)。
   预算有限时，**至少把 🔴 行全扫一遍**。

2. **比对**：把搜到的最新值和 watchlist 的「当前值」比。
   - 没变 → 跳过（可在 sweep-log 标「无变化」）。
   - 变了 / 出现新触发点事件 → 进入第 3 步。

3. **核真**（防污染纪律，不可省）：
   - **多源交叉**：≥2 个独立来源才算数；单一来源标「待核」。
   - **区分**【文献】(有据) vs【推演】(推断)；传闻标「传」。
   - 警惕**公关稿/促销口径**（产能≠在役、订单≠营收、目标≠实交、估值传闻≠收官）。

4. **落库**（确认属实后，四处同步更新）：
   - 对应**公司/环节档案**（速览卡 + 时间线 + 拨开迷雾，带日期信源）
   - **01-landscape 全表**的估值/状态列
   - **watchlist** 的「当前值 + 日期」
   - **根 README 更新日志**（一句话）

5. **留痕**：把本次 sweep 结果（变了什么 / 无变化 / 待核）追加到 [sweep-log.md](sweep-log.md)，标日期。

> 落库改完档案后，记得重建网页版：`node /Users/wchfting/Desktop/atom/_site/build.mjs`（让网站与档案同步）。

6. **报告 diff**：输出一段「本次变化摘要」——只讲**会改变判断**的变化（新轮落定、IPO 落地、盈利转向、重大订单、新入局），其余归档即可。

## 优先级原则
> 先核「会改变判断的」，再补增量细节。一条「宇树挂牌」比十条「某公司又发了个 Demo」重要得多。

## 边界 / 诚实
- 这是 pull-based 快照刷新，不是实时流；两次 sweep 之间发生的事，下次才会被抓到。
- 信源脏，sweep 抓的是「信号」，落库前要过测谎仪（[03-cases](../03-cases/README.md) 的自主 vs 遥操作纪律同样适用）。
- 拿不准就标「待核」，**宁可漏报，不可误报**。
